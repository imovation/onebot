# Onebot 起源与架构演进史 (Origin & Evolution)

本文档记录了 Onebot 架构诞生的原始需求、背景脉络以及核心架构思想的推导过程，以供后续维护者追溯其设计哲学。

## 1. 痛点：上下文污染与“精神分裂”的 Agent

Onebot 并非凭空设计，它脱胎于 `claw-swarm`（一个基于 OpenClaw 的编排代理项目）的重构过程。

在传统的 opencode 使用方式中，我们习惯于将所有“开发准则”、“核心红线”、“排错方法论”（如 deep-cure）一股脑地写进全局的 `opencode.json` 的 `instructions` 中。这导致了一个致命的问题：**上下文污染 (Context Poisoning)**。

无论是“日常写业务代码的开发者”，还是“仅作查询使用的终端用户”，底层的 LLM 始终被强制挂载着沉重的“开发包袱”。最终导致：
- 业务开发时，通用的底层排错方法论（deep-cure）经常越俎代庖，对简单的业务代码修改过度防御。
- 终端应用模式下，Agent 带着“开发视角”与用户沟通，产生角色错位。

## 2. 原始需求爆发：用户洞察与三态解耦

在定位到“双轨模式”失效的根源后，User 提出了极具前瞻性的顶层设计构想。

以下是触发本架构诞生的**核心原始需求记录（User 原话摘录）**：

> “我们从双轨模式失效推导出 claw-swarm 的定位问题。这是一个顶层设计或底层架构的重大问题。我觉得需要重新思考 claw-swarm 的顶层设计和底层架构。
>
> claw-swarm 的定位是基于 opencode 系统的 agent 这一点是肯定的，主要方向是从**上下文（系统提示词）工程**和**平台分层解耦**两个角度拆分：
> 
> - **上下文工程**：基于不同的使用场景或模式（比如开发、应用模式）上下文（系统提示词）按需加载。比如，开发模式需要读取跟开发相关的上下文（如 opencode 官方文档、Spec 驱动流程等），但这些在应用模式中是完全不需要的；再比如，我们今天在开发 deep-cure 过程中，原来治本原则也对我们造成了干扰甚至冲突。
> - **平台分层解耦**：从当前的 claw-swarm 中抽出通用的跟业务无关的部分（如 Spec 驱动流程，deep-cure等）套在原始 opencode 系统之外构建一套**中间层平台系统**。平台系统只解决通用问题，不解决具体业务问题。平台系统具备极强的通用性和可复用性。
> 
> 我希望你仅以 claw-swarm 的问题为例，必须跳出 claw-swarm，站在一个更高、更通用的角度，从【平台/业务】与【开发/应用】两个维度考虑，如何构建以 opencode 为基座的通用智能体解决方案。
>
> **注意：【平台/业务】与【开发/应用】两个维度非正交，平台应用=业务开发，也就是说平台的使命是用来开发业务的。组合后的场景是三个：平台开发 agent —> 平台应用即业务开发 agent —> 业务应用 agent。**”

## 3. 架构确立：The 3-Stage Agent Continuum

基于上述洞察，我们将这种“非正交的三场景”完美映射为 opencode 的物理架构，即 **Onebot 的三态演进模型**：

1. **平台开发 Agent (Platform Dev)**：造机器的机器。维护 `.opencode/`。
2. **业务开发 Agent (Biz Dev)**：即“平台应用”，在基建约束下生产业务代码。
3. **业务应用 Agent (App)**：纯粹面向用户的终端。

为了打破这三者的瀑布流孤岛，User 进一步提出了**“反哺闭环（Feedback Loop）”**：
- App 发现问题，写入 `REQUESTS.md`，交由 Biz Dev。
- Biz Dev 遇到基建瓶颈，写入 `.opencode/REQUESTS.md`，交由 Platform Dev。

至此，Onebot 不仅完成了物理隔离（瘦配置、胖角色、@file 注入），更成为了一个具备自组织、自进化能力的“智能体敏捷团队”。
