## 2026-04-11 第1次迭代 (方法论抽象)

**背景**：claw-swarm 项目 opencode 平台对齐诊断过程中，用户发现需要一种通用方法论。

**关键对话节点**：

> **用户**："我认可，但我还有补充：如果引入递归/迭代的思想如何优化这个 skill？"

> **用户**："我觉得整个 skill 应该是两条链路：诊断链和解决链。诊断链自下而上，通过识类和溯源层层诊断出每一层的问题；解决链自上而下，考虑到很多时候上层问题解决后下层问题就自然解决了，所以不要急于解决，首先思考每一层问题的解决方案，然后综合考虑要解决的问题和最终解决方案，最后按最终方案解决。"

> **AI**："我倾向 root-and-branch：它既是英语中'彻底解决'的固有表达，又恰好映射了这个方法的两条链路..."

> **用户**："名称我选 deep-cure，给出完整方案我确认。"

**意图**：

1. 渐进式按需加载 md 文档机制优化：opencode 不会自动解析 AGENTS.md 中的文件引用，需要用 opencode 原生机制（instructions、agents、skills）替代纯文本引用
2. 始终生效的规则（守门人、工程质量红线）应通过 opencode.json instructions 自动注入，而非依赖 Agent 自觉 Read
3. APP 模式的写权限限制应通过 opencode permission 机制机械强制，而非文本约束
4. Skill 目录应放在 opencode 搜索路径 (.opencode/skills/) 下
5. opencode.json 应配置 command、formatter、watcher.ignore 等原生特性
6. 项目定位声明：claw-swarm 是基于 opencode 的 agent，必须对齐平台规范（平台对齐准则）
7. 治本原则应扩展适用范围到所有层面（五问法），而非仅限代码 bug
8. 创建 deep-cure skill：将诊断过程抽象为通用方法论，核心是两条链路——诊断链（自下而上：治标→识类→溯源）和解决链（自上而下：治本药+退烧药），迭代收敛。用户补充：引入递归/迭代思想，修复后重新审视是否引入新问题；医生开药方比喻——治本药和退烧药都要开

**方法论核心确立**：
- 核心价值观："修复一个问题后，追问'同类还有哪些'和'为什么会产生这类问题'"
- 医生开药方比喻：抗病毒药（治本）+ 退烧药（治标）都要开

**期望结果**：
- .opencode/skills/deep-cure/SKILL.md 通用方法论 Skill
- Skill 按 skill-creator 规范编写，包含 YAML frontmatter、核心价值观、方法论、产出模板、案例参照

## 2026-04-11 第2次迭代 (诊断链递归分层)

**背景**：用户发现首轮方案中诊断链是扁平的，需要改为递归分层。

**关键对话节点**：

> **用户**："deep-cure 的诊断链底层逻辑可能有问题！我的想法是：分层的是问题，每一层的问题都要识类和溯源，问题溯源后就产生新的上层问题，然后再识类和溯源，直到结构性根因。你明白我的意思吗？"

> **用户**："你不只需要 SKILL.md，你还要更新原始意图、spec、最后才是按 spec 更新 SKILL.md。"

**意图**：

当前诊断链模型是扁平的：治标 → 识类(一次) → 溯源(直接跳到根因)。这跳过了中间层，压缩了诊断过程。

正确的模型应该是递归分层的：每一层的问题都要识类和溯源，溯源后产生新的上层问题，然后对新问题再做识类和溯源，直到触及结构性根因。

以 claw-swarm 案例为例：
- Level 1: 双轨失效 → 识类：5个同类 → 溯源：opencode不解析文件引用 → 产生新问题
- Level 2: 没用opencode原生机制 → 识类：5种场景 → 溯源：没查opencode规范 → 产生新问题
- Level 3: 没有规则要求查 → 识类：无 → 溯源：项目未声明自己是opencode agent → 产生新问题
- Level 4: 定位缺失 → 触及结构性根因 → 停止

**期望结果**：
- SPEC.md 诊断链模型从扁平改为递归分层
- SKILL.md 按新 SPEC 实现递归分层诊断链
- 案例部分用递归分层模型重写

**关联 Spec**：SPEC.md

## 2026-04-11 第3次迭代 (重新审视递归化)

**背景**：用户发现"重新审视"部分只是清单复查，不是真正的递归。

**关键对话节点**：

> **用户**："还有一个问题，'### 重新审视（迭代收敛）'这里是不是应该采用递归？"

> **用户**："我们刚刚分析的这些问题本身不怪你，但你应该能像我引导你一样，用我这样的思维链层层追问，直到找到根本问题或原因。你为什么没做到？如何能让你做到？"

> **AI**："根因是 `agents/core-rules.md` 里没有任何规则告诉 AI '要使用 opencode 原生机制'..."

> **用户**："你这不是举一反三，你这是头痛医头，脚痛医脚！... 我的意见是创建一个通用的 skill..."

**方法论升级**：
- 从"回到诊断链列出新问题"改为"对新问题重新走诊断链→解决链"
- 明确这是递归：每一轮发现的新问题都作为下一轮输入

**意图**：

"重新审视（迭代收敛）"部分应该采用递归——发现新问题后，应该对新问题再走一遍完整的诊断链→解决链，直到没有新问题。当前描述只是"回到诊断链"列出新问题，不是真正的递归。

**期望结果**：
- SPEC.md 重新审视改为递归应用完整方法
- SKILL.md 同步更新
