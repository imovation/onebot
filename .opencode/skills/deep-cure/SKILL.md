# Deep Cure — 深度诊断与根治

## 核心价值观

> 修复一个问题后，追问"同类还有哪些"和"为什么会产生这类问题"。

这个 Skill 不教你修一个 bug — 它教你修一类 bug，并消灭让这类 bug 反复出现的土壤。

## 何时使用

- 用户明确要求深度诊断时
- 用户说"举一反三"、"还有类似问题吗"、"根本原因是什么"、"治本了吗"时
- 你完成一个修复后，直觉告诉你可能还有同类问题潜伏时
- 主动提议：当你发现某个修复只是在链路末端打补丁，而不是从源头解决时

## 何时不用

- 简单明确的操作指令（如"运行测试"、"格式化这个文件"）
- 用户只需要修一个具体 bug 且明确表示不需要深入分析时

## 方法论：递归诊断链 + 双链迭代

### 诊断链（递归分层，自下而上）

诊断链不是一条直线，而是递归的分层结构。每一层问题都要识类和溯源，溯源的答案产生新的上层问题，对新问题再做识类和溯源，直到触及结构性根因。

```
Level 1 (症状层):
  问题 → 识类(同类清单, 根因维度) → 溯源("为什么？")
    ↓ 答案 = 新问题
Level 2 (中间层):
  新问题 → 识类(同类清单, 根因维度) → 溯源("为什么？")
    ↓ 答案 = 又一个新问题
Level 3 (更上层):
  ...
    ↓ 直到触及结构性根因
Level N (根因层):
  结构性根因 → 停止追问
```

关键：每一层的识类维度不同。不是一次识类后直接跳到根因，而是每层有自己的根因维度和同类清单，逐层上升。

三类结构性根因（追问到此即停）：

| 类型 | 含义 | 判断标准 |
|------|------|---------|
| **定位缺失** | 项目没给自己明确定位 | 如果项目声明了"我们是 X 平台的 Y"，这类问题就不会发生 |
| **假设过时** | 项目基于某个不再成立的假设 | 如果项目验证了这个假设，这类问题就不会发生 |
| **流程缺口** | 缺少某个必要的检查环节 | 如果流程中有这一步检查，这类问题就不会发生 |

### 解决链（递归分层，自上而下 — 像开药方）

解决链与诊断链的分层结构一一对应。从诊断链的最顶层（结构性根因）开始，逐层向下审视：

**Level N（根因层）— 抗病毒药**

修复结构性根因，消除病因，防止同类问题复发。

**Level N-1, N-2, ... Level 1 — 退烧药检查**

对每一层问：修了上层后，这层的症状是否仍需独立修复？

- 如果上层修复后本层症状自然消失 → 确认，跳过
- 如果本层症状不会自愈 → 必须开退烧药，确保当下不恶化

**综合方案**

合并所有层级的修复项，按依赖顺序排列，一次执行。

**核心原则：治本确保未来不复发，治标确保当下不恶化。两者都不可省略。**

### 重新审视（递归迭代）

修复执行完后，不要宣布完成。对可能引入的新问题，重新应用完整的 deep-cure 过程：

1. 修复是否引入了新问题？如果有，对新问题重新走诊断链→解决链
2. 项目状态变化后，之前的诊断结论是否仍然成立？如果有变化，重新评估
3. 有些问题在修复根因后可能自然消失 → 确认，跳过

这不是简单的清单复查，而是递归：每一轮发现的新问题都作为下一轮 deep-cure 的输入，走完整的诊断→解决流程。

每轮迭代产生的新问题严格少于上一轮（因为治本药防止了同类复发），所以必然收敛。无新问题浮现时结束。

## 产出模板

每轮迭代输出以下结构：

```markdown
## 诊断链

### Level 1: <问题简述>
- 识类：根因维度 = <从问题抽象出的根因类别>
  - 同类清单：
    1. <实例 A>
    2. <实例 B>
    3. <实例 N>
- 溯源：为什么？→ <答案> → 产生新问题

### Level 2: <上层问题简述>
- 识类：根因维度 = <新的根因类别>
  - 同类清单：
    1. <实例>
    2. <实例>
- 溯源：为什么？→ <答案> → 产生新问题

...

### Level N: <结构性根因>
- 类型：<定位缺失 / 假设过时 / 流程缺口>
- 描述：<详细说明>

## 解决链

| 层级 | 问题性质 | 修复方案 | 能否仅靠治本解决？ |
|------|---------|---------|-----------------|
| Level N (根因) | 治本 | <方案> | — |
| ... | 治本兼有/治标 | <方案> | <能/不能，原因> |
| Level 1 (症状) | 治标 | <方案> | 不能，<原因> |

## 综合方案

<从根因到症状的完整修复方案，按执行顺序列出>

## 重新审视

- 修复是否引入新问题？<是/否>
- 如是，新问题：<描述>
- 如继续迭代，对新问题重新走诊断链→解决链，产出与本模板相同结构的完整报告
- 结论：<继续迭代 / 收敛结束>
```

## 案例：claw-swarm opencode 对齐诊断

本 Skill 源于一个真实案例，包含两轮递归迭代。

### 第一轮

诊断链：
- **Level 1**: 双轨模式失效 → 识类(5个同类，根因维度="应始终生效的规则未被自动加载") → 溯源:因为 opencode 不解析文件引用 → 产生新问题
- **Level 2**: 没用 opencode 原生机制 → 识类(5种场景，根因维度="开发时没有查阅平台规范") → 溯源:因为没有规则要求查 → 产生新问题
- **Level 3**: 没有规则要求查平台规范 → 识类(没有同类，这是唯一缺失) → 溯源:因为项目从未声明自己是 opencode agent → 产生新问题
- **Level 4**: 定位缺失 → **触及结构性根因** → 停止

解决链：
- **Level 4（根因层）抗病毒药**：SYSTEM_SPEC 新增平台对齐准则
- **Level 3 退烧药**：core-rules 新增"opencode 原生优先"
- **Level 2 退烧药**：opencode.json 配置、.opencode/ 目录、Skill 迁移
- **Level 1 退烧药**：AGENTS.md @file 指令、meta.md 更新、SYSTEM_SPEC §三 更新

重新审视：修复后发现 SYSTEM_SPEC §三 过时、meta.md 冗余 → 进入第二轮

### 第二轮（递归迭代）

诊断链：
- **Level 1**: 6 处双重声明和冗余（gatekeeper 模式选择与 opencode agent 重复、权限限制与 permission 重复、git 闭环与 dev-mode.md 重复、Spec 驱动与 core-rules 重复、opencode 原生优先在两处声明、加载时机声明冗余） → 识类(6个同类，根因维度="同一约束被多个机制双重声明") → 溯源:因为引入新机制后没有清理旧机制中被替代的部分 → 产生新问题
- **Level 2**: 只加不减的修复思维 → 识类(这是普遍倾向，无更多同类) → 溯源:因为没有"替换清理"的判断准则 → 产生新问题
- **Level 3**: 流程缺口 → **触及结构性根因** → 停止

解决链：
- **Level 3（根因层）抗病毒药**：core-rules.md 新增第 7 条"替换清理准则"
- **Level 2 退烧药**：不会自愈，需逐案判断哪些旧内容被新机制替代
- **Level 1 退烧药**：精简 gatekeeper.md、去重 dev-mode.md、删除冗余加载声明、修正编号

重新审视：修复后核心编号不连续、SYSTEM_SPEC 守门人行数过时 → 修正 → 无新问题 → **收敛结束**

注意：此案例旨在帮助理解方法论，不要将 Skill 限定在特定语境中。deep-cure 适用于任何类型的深度诊断。
