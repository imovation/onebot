## 1. 核心意图 (Core Intent)
提供一套递归分层的根因诊断标准作业程序 (SOP)。强迫 Agent 放弃“头痛医头”的补丁式思维，通过溯源直击结构性根因（定位缺失、假设过时、流程缺口），产出“标本兼治”的解决方案。

## 2. 边界约束 (Constraints)
- 🚫 **诊断禁区**：严禁在未触及“结构性层级（机制、架构、流程）”前停止追问为什么（至少 5 问）。
- 🚫 **给药禁区**：严禁只开“抗病毒药（改机制）”而不开“退烧药（修现状代码）”，两者缺一不可。
- 🚫 **迭代禁区**：严禁在执行修复后直接宣布完成，必须执行递归的“重新审视”。

## 3. 状态契约 (State Contract)
- **触发条件 (Inputs)**：用户质问“根本原因是什么”、“治本了吗”，或开发者陷入反复报错的泥潭。
- **预期副作用 (Side Effects)**：产出标准化的多层级《深度诊断与根治报告》，并伴随底层机制的代码/架构修改。

## 4. 验收标准 (Acceptance Criteria)
- [ ] 标准 1：最终交付的修复方案必须包含针对架构规范或机制流程的调整，以杜绝同类问题的未来复发。
- [ ] 标准 2：文档必须包含明确的【诊断链】、【解决链】和【重新审视】三大模块。

## 5. 方法论细节 (Implementation Details)

### 方法论：递归诊断链 + 双链迭代

### 诊断链（递归分层，自下而上）

诊断链不是扁平的三步，而是递归的分层结构：每一层问题都要识类和溯源，溯源的结果产生新的上层问题，对新问题再识类和溯源，直到触及结构性根因。

```
Level N (症状层):
  问题 → 识类(同类清单) → 溯源("为什么？")
    ↓ 溯源答案 = 新问题
Level N+1 (中间层):
  新问题 → 识类(同类清单) → 溯源("为什么？")
    ↓ 溯源答案 = 又一个新问题
Level N+2 (更上层):
  ...
    ↓ 直到触及结构性根因
Level 最终:
  结构性根因(定位缺失/假设过时/流程缺口) → 停止追问
```

三类结构性根因（追问到此即停）：

| 类型 | 含义 | 示例 |
|------|------|------|
| **定位缺失** | 项目没给自己明确定位 | "我们是 X 平台的 agent"从未声明 |
| **假设过时** | 项目基于某个不再成立的假设运行 | "AI 会自动解析文件引用" |
| **流程缺口** | 缺少某个必要的检查环节 | "开发前先查平台文档" |

关键区别：每一层的识类维度不同。不是一次识类后直接跳到根因，而是每层有自己的根因维度，逐层上升。

### 解决链（递归分层，自上而下开药方）

解决链与诊断链的分层结构一一对应。从诊断链的最顶层（结构性根因）开始，逐层向下审视：

1. **Level N（根因层）— 抗病毒药**：修复结构性根因，消除病因，防止同类问题复发
2. **Level N-1, N-2, ... Layer 1 — 退烧药检查**：对每一层问：修了上层后，这层的症状是否仍需独立修复？
   - 如果上层修复后本层症状自然消失 → 跳过
   - 如果本层症状不会自愈 → 开退烧药，确保当下不恶化
3. **综合方案**：合并所有层级的修复项，按依赖顺序排列，一次执行

核心原则：**治本确保未来不复发，治标确保当下不恶化。两者都不可省略。**

### 重新审视（递归迭代）

修复执行完后，不要宣布完成。对可能引入的新问题，重新应用完整的 deep-cure 过程：

1. 修复是否引入了新问题？如果有，对新问题重新走诊断链→解决链
2. 项目状态变化后，之前的诊断结论是否仍然成立？如果有变化，重新评估
3. 有些问题在修复根因后可能自然消失 → 确认，跳过

这不是简单的清单复查，而是递归：每一轮发现的新问题都作为下一轮 deep-cure 的输入，走完整的诊断→解决流程。

每轮新问题严格少于上一轮（因为治本药防止了同类复发），所以必然收敛。无新问题浮现时结束。

## 产出模板

每轮迭代按以下结构输出：

对诊断链的每一层：
- 问题：当前层的问题描述
- 识类：根因维度 + 同类清单
- 溯源：为什么？→ 答案 = 上一层的问题

然后依次记录直到触及结构性根因。

最后：
- 解决链 — 各层级问题性质、修复方案、能否仅靠治本解决的评估
- 综合方案 — 完整修复方案，按执行顺序列出
- 重新审视 — 是否引入新问题。如继续迭代，对新问题重新走诊断链→解决链，产出完整报告

## 触发条件

- 用户明确要求深度诊断
- 用户说"举一反三"、"还有类似问题吗"、"根本原因是什么"
- 用户对修复表示怀疑（"这个治本了吗？"）

不触发于简单操作指令（如"运行测试"、"格式化文件"）。

## 来源

本 Skill 源于 claw-swarm 项目的一项实际诊断，包含两轮递归迭代：

第一轮诊断链：
- Level 1: 双轨模式失效 → 识类(5个同类，根因维度="应始终生效的规则未被自动加载") → 溯源:因为 opencode 不解析文件引用 → 产生新问题
- Level 2: 没用 opencode 原生机制 → 识类(5种场景) → 溯源:因为开发时没查 opencode 规范 → 产生新问题
- Level 3: 没有规则要求查规范 → 识类(无同类，这是唯一缺失) → 溯源:因为项目从未声明自己是 opencode agent → 产生新问题
- Level 4: 定位缺失 → 触及结构性根因 → 停止

第二轮诊断链（递归迭代，审视第一轮修复引入的问题）：
- Level 1: 6处双重声明和冗余 → 识类(6个同类，根因维度="同一约束被多个机制双重声明") → 溯源:因为引入新机制后没有清理旧机制中被替代的部分 → 产生新问题
- Level 2: 只加不减的修复思维 → 识类(普遍倾向) → 溯源:因为没有替换清理判断准则 → 产生新问题
- Level 3: 流程缺口 → 触及结构性根因 → 停止
